# Ссылки: админка и биллинг

## Админка

Доступ только для пользователей с `is_admin: true`. Вход тем же логином, что и на основном сайте.

| Среда      | Ссылка |
|------------|--------|
| Локально   | http://localhost:12000/admin |
| По LAN     | http://ВАШ_IP:12000/admin |
| Продакшен  | https://pingwin.pro/admin |

- **Список пользователей:** `/admin/users` (поиск, пагинация)
- **Карточка пользователя:** `/admin/users/{id}` — профиль, подписки, счета, триал, выдача подписки
- **Догрузка результатов:** `POST /api/v1/matches/backfill-results` (только админ) — создание MatchResult из MatchScore для зависших матчей, сброс попыток у старых без результата, дозапрос к BetsAPI

## Биллинг (оплата подписок)

### Для пользователей (фронт)

| Страница        | Ссылка (локально)           | Ссылка (продакшен)        |
|-----------------|-----------------------------|----------------------------|
| Тарифы / оплата | http://localhost:12000/pricing | https://pingwin.pro/pricing |
| После оплаты    | http://localhost:12000/pricing?paid=1 | https://pingwin.pro/pricing?paid=1 |

Кнопка «Оплатить» на странице тарифов создаёт платёж в ЮKassa и перенаправляет на защищённую страницу оплаты; после успешной оплаты пользователь возвращается на `BILLING_RETURN_URL` (в .env: `BILLING_RETURN_URL=https://pingwin.pro/pricing?paid=1`).

### API бэкенда (для справки)

| Назначение           | Метод и путь |
|----------------------|--------------|
| Создать платёж       | `POST /api/v1/billing/checkout` |
| Список счетов юзера  | `GET /api/v1/billing/invoices` |
| Вебхук ЮKassa        | `POST /api/v1/billing/webhook/yookassa` |

Полный URL вебхука для кабинета ЮKassa: `https://ваш-домен-бэкенда/api/v1/billing/webhook/yookassa` (тот же домен, на котором висит API, например за nginx).

## 502 Bad Gateway при запросах к API / WebSocket

502 отдаёт **прокси** (nginx/caddy), когда он не получает ответ от бэкенда. Типичные причины:

1. **Бэкенд не запущен**  
   На сервере должен слушать порт бэкенда (в примере nginx — `127.0.0.1:12001`). Проверка:
   - `curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:12001/health` — должен вернуть `200`.
   - Если используете Docker/systemd — убедиться, что контейнер/сервис бэкенда в состоянии running.

2. **Бэкенд падает при старте**  
   Смотреть логи процесса (Docker: `docker logs <backend_container>`, systemd: `journalctl -u <backend_service> -n 100`). Частые причины:
   - ошибка подключения к БД (неверный `DATABASE_URL`, PostgreSQL не запущен);
   - не заданы обязательные переменные в `.env`;
   - ошибка импорта (например, после обновления кода).

3. **Таймаут прокси**  
   Если бэкенд отвечает слишком долго, nginx по умолчанию обрывает соединение и отдаёт 502. Для долгих запросов можно увеличить в конфиге nginx, например:
   - `proxy_connect_timeout 60s; proxy_send_timeout 60s; proxy_read_timeout 60s;` в `location /api/`.

4. **Неверный upstream**  
   В конфиге прокси для `location /api/` и `location /ws` должен быть указан правильный хост и порт бэкенда (см. `docs/nginx-pingwin.example.conf`).

**Быстрая проверка с сервера:**  
`curl -s http://127.0.0.1:12001/health` — если бэкенд жив, ответ `{"status":"ok"}`.

## 500 Internal Server Error

Если запросы к API возвращают **500** (а не 502), ошибка происходит уже в приложении. В теле ответа теперь всегда есть поля **`error_type`** и **`error_detail`** — откройте вкладку Network в DevTools, выберите запрос с 500 и посмотрите Response.

Частые причины:

1. **Нет таблицы в БД** — например `relation "user_signal_deliveries" does not exist` или `column users.trial_until does not exist`. Нужно применить миграции:
   ```bash
   cd backend && alembic upgrade head
   ```
2. **Ошибка подключения к БД или Redis** — в `error_detail` будет сообщение от драйвера.
3. **Неверные или отсутствующие переменные в `.env`** — проверьте `DATABASE_URL`, `SECRET_KEY` и др.

После исправления перезапустите бэкенд.
