# BetsAPI: как мы получаем данные

Документ описывает: какие запросы делаем, что из какого ответа берём, куда пишем в БД, как определяем завершение матча, как обновляем линию и как получаем коэффициенты.

---

## 1. Режимы и расписание

| Режим       | Задача Celery                          | Интервал (по умолчанию) | Назначение |
|------------|----------------------------------------|--------------------------|------------|
| **full**   | `fetch-betsapi-table-tennis-full`      | 600 сек (10 мин)         | Линия + лайв: списки, детали (view), коэффициенты по линии один раз; по лайву — только для матчей с изменившимся счётом. |
| **live**   | `fetch-betsapi-table-tennis-live`     | 3 сек                    | Список inplay → обновление счёта; коэффициенты запрашиваем **сразу только по матчам, где изменился счёт** (без отдельного таймера). |

Отдельная задача **live_odds** по расписанию отключена: коэффициенты лайва подтягиваются в цикле live по факту изменения счёта.

Конфиг: `LIVE_POLL_INTERVAL_SECONDS`, `PREMATCH_POLL_INTERVAL_SECONDS` в `.env`.

### Сколько запросов в час (при текущей реализации)

Дефолт: live каждые 3 с, full каждые 600 с. Коэффициенты лайва — только при изменении счёта (N запросов event/odds на цикл, где N = число матчей с изменившимся счётом).

| Режим | Запусков в час | Запросов за 1 запуск | В час |
|-------|----------------|----------------------|-------|
| **live** | 1200 | 1 (inplay) + **S** (event/odds только по матчам с изменившимся счётом), S ≥ 0 | **1200 + 1200×S_avg** |
| **full** | 6 | 2 + ceil((U+I)/10) (view) + (**S_inplay** + U_new) (odds), S_inplay = inplay с изменением счёта | **6 × [2 + view + odds]** |

Обозначения: U = кол-во upcoming, I = inplay, U_new = upcoming без line-odds в БД; S_avg — среднее число матчей с изменением счёта за один цикл live (обычно 0–2).

**Итого в час:** 1200 + (запросы event/odds по факту изменения счёта в live) + 6×(full). Без изменений счёта по лайву — порядка **1200 + ~150 (full)** ≈ **1350**; при активных матчах — больше, но только за счёт реальных изменений.

## 2. Какие запросы делаем

### Режим **live** (каждые 3 сек)

| № | Запрос | Параметры | Что получаем |
|---|--------|-----------|--------------|
| 1 | `GET /v3/events/inplay` | `sport_id`, `token` | Список событий в лайве (и только что завершённых). В каждом: `id`, `time`, `time_status`, `league`, `home`, `away`, **`ss`**, **`scores`** (счёт по сетам уже в ответе). |
| 2..N+1 | `GET /v2/event/odds?event_id=<id>` | по одному id | **Только по матчам, у которых изменился счёт по сетам** (сравнение с БД). N = число таких матчей в текущем цикле. |

**Итого за цикл:** 1 запрос inplay; если счёт изменился — дополнительно по 1 запросу event/odds на каждый такой матч. Без изменений счёта — 1 запрос.

---

### Режим **full** (каждые 10 мин)

**Фаза 1 — только списки:**

| № | Запрос | Параметры | Что получаем |
|---|--------|-----------|--------------|
| 1 | `GET /v3/events/upcoming` | `sport_id`, `token` | Список событий в линии (до начала). |
| 2 | `GET /v3/events/inplay` | `sport_id`, `token` | Список событий в лайве (и недавно завершённых). |

Список **ended** не запрашиваем — завершённые определяем по inplay (`time_status`) и по исчезновению из списков.

**Фаза 2 — детали и коэффициенты:**

- По БД определяем, у каких **upcoming** уже есть хотя бы один снимок коэффициентов с `phase='line'` (или `phase IS NULL`).
- По ответу inplay определяем, у каких матчей **изменился счёт по сетам** относительно БД.
- Запрашиваем **view** только для событий из **upcoming + inplay** (не для ended).
- Запрашиваем **odds** для:
  - только тех **inplay**, у которых изменился счёт по сетам;
  - только тех **upcoming**, у которых ещё нет line-коэффициентов в БД.

| № | Запрос | Параметры | Что получаем |
|---|--------|-----------|--------------|
| 3 | `GET /v1/event/view?event_id=id1,id2,...,id10` | батчами по 10 id | Доп. поля: `ss`, `extra`, `timeline` и т.д. Мержим в объект события. |
| 4..M | `GET /v2/event/odds?event_id=<id>` | по одному id | Коэффициенты по рынкам (формат v2 для НТ: `results.odds` → рынки 92_1, 92_2, 92_3). |

**Итого за full:** 2 (списки) + ceil((upcoming+inplay)/10) (view) + (inplay с изменением счёта + upcoming без odds в БД) (odds).

## 3. Что из какого запроса берём и куда пишем

### 3.1 Списки: upcoming / inplay

**Источник:** `GET /v3/events/upcoming`, `GET /v3/events/inplay` → поле `results` (массив событий).

**Из каждого события берём:**

| Поле API   | Куда пишем | Таблица/модель |
|------------|------------|-----------------|
| `id`       | Идентификатор события у провайдера | `Match.provider_match_id` |
| `time` (unix) | Время начала | `Match.start_time` |
| `league.id`, `league.name`, `league.cc` | Лига | `League` (get_or_create), `Match.league_id` |
| `home.id`, `home.name` | Хозяин | `Player` (get_or_create), `Match.home_player_id` |
| `away.id`, `away.name` | Гость | `Player` (get_or_create), `Match.away_player_id` |
| `_source` (мы добавляем: "upcoming" / "inplay") | — | Определяет, обновлять ли status и счёт (см. ниже). |

**Дополнительно из inplay:** счёт по сетам (`scores`, `ss`) приходит **в том же ответе списка** inplay — отдельный запрос на каждый матч для сетов не делается.

| Поле API   | Куда пишем | Таблица/модель |
|------------|------------|-----------------|
| `time_status` | Определение лайв/завершён | см. раздел 4 |
| `ss`       | Счёт по сетам (строка) | Парсим → `MatchScore` (set_number, home_score, away_score) |
| `scores`   | Счёт по сетам (dict: "1"→{home, away}, "2"→...) | То же → `MatchScore` (приоритет над `ss` при наличии). |

**Как пишем:** для каждого события делаем `get_or_create` по `(provider, provider_match_id)` для League, Player, Match. Для inplay/ended обновляем `Match.status` и перезаписываем счета в `MatchScore` (delete по match_id, затем insert по сетам).

---

### 3.2 Детали: event/view

**Источник:** `GET /v1/event/view?event_id=id1,id2,...` → `results` (dict event_id → объект или list при одном id).

**Используем:** мержим в уже полученное событие из списка. Берём недостающие поля: `ss`, `score`, `extra`, `timeline` и т.д. Дальше те же поля, что и из списка, идут в нормалайзер (в т.ч. `ss`/`scores` → `MatchScore`).

View запрашиваем **только для upcoming + inplay**; по завершённым (ended) view не вызываем.

---

### 3.3 Коэффициенты: event/odds

**Источник:** `GET /v2/event/odds?event_id=<id>` → `results.odds` (для наст. тенниса: dict рынок → список снимков).

**Формат v2 для НТ:** ключи `92_1`, `92_2`, `92_3`; значение — список элементов `{ home_od, away_od, ss, add_time }`. Берём снимок с максимальным `add_time` по каждому рынку; значения `"-"` пропускаем.

**Куда пишем:** после преобразования в формат «букмекер → рынки → исходы» (в коллекторе) нормалайзер пишет в **OddsSnapshot**:

| Из ответа (после нормализации) | Поле в БД |
|-------------------------------|-----------|
| Букмекер (у нас один виртуальный "b365") | `OddsSnapshot.bookmaker` |
| Рынок (92_1, 92_2, 92_3) | `OddsSnapshot.market` |
| Исход (home / away) | `OddsSnapshot.selection` |
| Коэффициент | `OddsSnapshot.odds`, `implied_probability` |
| Фаза: линия или лайв | `OddsSnapshot.phase` = `"line"` (upcoming) или `"live"` (inplay) |

Один запрос event/odds даёт один «снимок» по матчу; мы **добавляем** строки в `odds_snapshots`, не удаляем старые — накапливается история.

## 4. Как считаем, что матч завершён

1. **По полю в ответе inplay**  
   В объекте события из `GET /v3/events/inplay` смотрим:
   - `time_status` in (`"3"`, `"100"`, 3, 100) → матч завершён;
   - или `status` in (`"ended"`, `"closed"`, `"finished"`) → матч завершён.

   В нормалайзере: `_is_event_ended(event)`. Для таких событий выставляем `Match.status = FINISHED`, пишем счёт в `MatchScore`, создаём одну запись в **MatchResult** (winner_id, final_score, finished_at), если её ещё нет.

2. **По исчезновению из списков**  
   После нормализации вызываем обновление: все матчи со статусом **LIVE** и с `provider_match_id`, которых **нет** в множестве текущих id из **upcoming** и **inplay**, помечаем как **FINISHED**.  
   То есть если матч раньше был в inplay, а в текущем ответе его уже нет ни в upcoming, ни в inplay — считаем его завершённым и больше не запрашиваем по нему view/odds.

Отдельный запрос **events/ended** мы не используем.

---

## 5. Как матчи из линии переходят в лайв

Один и тот же матч на стороне BetsAPI сначала попадает в **upcoming**, а после начала — в **inplay**. У нас он один и тот же по `provider_match_id`.

- **Первый раз** матч приходит в списке **upcoming** (режим full или при следующем full после появления в линии). Нормалайзер делает `_get_or_create_match` → создаётся запись **Match** со статусом **SCHEDULED** (значение по умолчанию при создании). Коэффициенты по линии можем подтянуть в том же цикле full (если запрашиваем odds для этого upcoming).
- **Когда матч начался**, BetsAPI перекладывает его из списка upcoming в список inplay. В следующем опросе (режим **live** каждые 3 сек или **full** каждые 10 мин) этот матч приходит уже в **inplay** с тем же `id`. Нормалайзер находит существующий Match по `provider_match_id`, видит `_source == "inplay"` и выставляет **Match.status = LIVE**, обновляет **MatchScore** по полям `scores`/`ss`.

Итого: переход «линия → лайв» — это не перенос между таблицами, а **обновление статуса одного и того же Match**: сначала SCHEDULED (из upcoming), затем LIVE (когда тот же id пришёл в ответе inplay). Отдельного запроса «начался матч» нет — мы просто обрабатываем списки; в каком списке пришёл id, такой статус и выставляем.

---

## 6. Как завершённые матчи попадают в результаты

Под «результатами» имеется в виду отображение итога матча: победитель, финальный счёт, время окончания. В БД это одна запись **MatchResult** на матч и статус **Match.status = FINISHED**.

- Когда обрабатываем событие из **inplay** и `_is_event_ended(event)` возвращает True (`time_status` 3/100 или `status` ended/closed/finished), мы:
  1. Ставим **Match.status = FINISHED**.
  2. Пишем счёт по сетам в **MatchScore** (как и для лайва).
  3. Если записи **MatchResult** для этого матча ещё нет — создаём её: `match_id`, `winner_id` (home/away по количеству выигранных сетов), `final_score` (строка счёта), `finished_at`.

- Когда матч **исчез из списков** (его `provider_match_id` нет в `current_event_ids` = upcoming ∪ inplay), мы помечаем все такие LIVE-матчи как **FINISHED**. Запись **MatchResult** при этом не создаётся в этом шаге (мы не знаем финальный счёт из ответа). Её можно создать позже, если появится источник с итогом, либо оставить без результата — в коде сейчас MatchResult создаётся только при обработке события из inplay с признаком завершения.

Итого: завершённый матч — это **Match** с **status = FINISHED** и при наличии — одна связанная запись **MatchResult**. «Переход в результаты» = обновление статуса матча на FINISHED + создание (один раз) MatchResult при наличии данных о счёте в ответе inplay.

## 7. Как обновляем линию

- **Линия** — это события из **upcoming** (список приходит в режиме **full** каждые 10 мин).
- В режиме full мы:
  1. Запрашиваем `upcoming` + `inplay` (списки).
  2. Запрашиваем **view** для всех upcoming + inplay (детали).
  3. Запрашиваем **odds** только для тех upcoming, у которых в БД ещё нет ни одного OddsSnapshot с `phase='line'` (или `phase IS NULL`).

Таким образом линия обновляется раз в 10 минут: обновляются списки матчей, детали и один раз подтягиваются коэффициенты по каждому матчу линии; повторно odds по линии не дергаем.

Новые матчи в линии появляются при следующем запуске full (при следующем запросе upcoming).

---

## 8. Как получаем коэффициенты

- **По линии (прематч):**  
  В режиме **full** запрашиваем event/odds только по тем upcoming-событиям, у которых в БД ещё нет line-снимка (проверка по `Match` + `OddsSnapshot` с `phase='line'` или `phase IS NULL`). Ответ парсим (v2 формат НТ), пишем в **OddsSnapshot** с `phase='line'`. Для каждого такого матча делаем это один раз за цикл; в следующие циклы он уже в «upcoming_with_odds» и по нему odds в full не запрашиваем.

- **По лайву:**  
- **По лайву:** в режиме **live** (каждые 3 сек) после получения списка inplay сравниваем счёт по сетам с БД; только у матчей с изменением счёта запрашиваем `GET /v2/event/odds?event_id=<id>`. Парсим `results.odds` (92_1, 92_2, 92_3), пишем в **OddsSnapshot** с `phase='live'`. В режиме **full** по inplay запрашиваем event/odds только для матчей с изменившимся счётом.

- **По завершённым:**  
  View и odds по завершённым матчам не запрашиваем (их id не попадают в списки для view и для odds).

Итог: линия — коэффициенты подтягиваем один раз (при первом появлении в upcoming с пустым line-odds в БД). Лайв — коэффициенты обновляем **только при изменении счёта по сетам** (в цикле live и в full только по таким матчам). Завершённые — не обновляем.

## 9. Сводная схема запросов и записей

```
LIVE (каждые 3 сек):
  GET /v3/events/inplay
    → события с _source=inplay
    → сравнение счёта с БД → список id с изменением счёта
    → при наличии: GET /v2/event/odds по каждому такому id
    → Normalizer: Match.status=LIVE/FINISHED, MatchScore, OddsSnapshot (phase=live) только по матчам с odds
    → current_event_ids = id из inplay

FULL (каждые 10 мин):
  GET /v3/events/upcoming  → события _source=upcoming
  GET /v3/events/inplay    → события _source=inplay
  → БД: какие upcoming уже имеют OddsSnapshot (phase=line или NULL); какие inplay имеют изменение счёта
  GET /v1/event/view батчами по (upcoming + inplay)
  GET /v2/event/odds по каждому inplay с изменением счёта + по каждому upcoming без line-odds
    → Normalizer: League, Player, Match, MatchScore, MatchResult (если ended), OddsSnapshot (phase=line или live)
  → current_event_ids = upcoming ∪ inplay (для пометки исчезнувших LIVE как FINISHED)
```
